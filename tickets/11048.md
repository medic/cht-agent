# Description

There is an existing TODO in `admin/src/js/services/version.js` about replacing the current version-parsing regex once support for older Node versions is dropped. The project is now on Node 22, so the regex can be modernized to use named capture groups.

The current implementation relies on numbered array indexes (`versionMatch[1]`, `versionMatch[2]`, ...), which is harder to read and easier to break when the regex changes. A commented example already shows the intended named-group version:

```
/^(?<major>\d+)\.(?<minor>\d+)\.(?<patch>\d+)(?<featureRelease>-FR(?:-\w+)+)?(?:-beta\.(?<beta>\d+))?$/
```

The migration should:

- Replace the current regex with the named-capture-group version.
- Update the parsing logic to use `versionMatch.groups.major`, `versionMatch.groups.minor`, etc.
- Keep the existing parsing behavior unchanged.

## Requirements

- Replace the active regex in `admin/src/js/services/version.js` with the named-capture-group version that already appears in the file as a comment.
- Update all consumers of the regex in that file to read fields via `versionMatch.groups.<name>` instead of numeric indexes.
- The output shape and semantics of the version parser remain unchanged.

## Acceptance Criteria

- The regex in `admin/src/js/services/version.js` uses named capture groups for `major`, `minor`, `patch`, `featureRelease`, and `beta`.
- The TODO comment about awaiting newer Node versions is removed (or replaced by an explanatory comment if anything else justifies retention).
- All existing tests in the version parsing test suite continue to pass without modification of the assertions.

## Constraints

- No change in parser behavior or return shape.
- No new dependencies.
- No version-number string format changes; the regex must still match the same strings it matched before.
