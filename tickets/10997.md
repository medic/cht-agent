# Description

When a user is redirected to the login page from a URL containing query parameters, those query parameters are lost after successful authentication. The user is redirected back to the correct path but without the original query string.

## Steps to Reproduce

1. While unauthenticated, attempt to access a URL with query parameters, e.g. `/reports?search=malaria&sort=date#selected`.
2. Get redirected to the login page.
3. Log in successfully.
4. Observe the redirect URL.

## Expected Behavior

User is redirected to `/reports?search=malaria&sort=date#selected` — the full original URL is preserved.

## Actual Behavior

User is redirected to `/reports#selected` — query parameters are stripped.

## Root Cause

In `api/src/controllers/login.js`, the `resolveUrl()` helper correctly preserves query parameters:

```js
const { pathname, search, hash } = resolvedUrl;
return pathname + search + hash;
```

However, `sanitizeRequestedRedirect()` re-parses the resolved URL and drops `parsed.search`:

```js
const parsed = new URL(resolved, 'resolve://');
const path = parsed.pathname + (parsed.hash || '');
```

The `search` is omitted during the sanitize step, so it never reaches the final redirect URL.

## Requirements

- After successful login, the post-authentication redirect URL preserves any query parameters that were present in the original request URL.
- Sanitization rules continue to reject open-redirect attacks and external hosts; only the same-origin path + search + hash is preserved.
- Existing tests for sanitization (open-redirect prevention, host validation) continue to pass.

## Acceptance Criteria

- After login from `/reports?search=malaria&sort=date#selected`, the user lands on `/reports?search=malaria&sort=date#selected`.
- After login from `/reports#selected` (no query), the user lands on `/reports#selected` (unchanged behavior).
- After login from a malicious `redirect=https://evil.example.com/path?foo=bar` URL, the redirect is rejected by sanitization just as before.
- Test coverage added in `api/tests/mocha/controllers/login.spec.js` for the query-preservation case.

## Constraints

- Cannot weaken `sanitizeRequestedRedirect`'s security guarantees.
- No external dependencies added.
- Fix is in `api/src/controllers/login.js` and its tests; do not modify unrelated controllers.

## Technical Context

**Components:**
- `api/src/controllers/login.js`
- `api/tests/mocha/controllers/login.spec.js`

## Suggested Fix

Include `parsed.search` in the path construction:

```js
const path = parsed.pathname + (parsed.search || '') + (parsed.hash || '');
```
