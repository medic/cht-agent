## Problem

Scheduled messages were generated when a report was received, not when they were sent. This meant:

1. If patient data changed between report receipt and message send time, messages contained outdated information
2. Translation values were stale (could be days old)
3. Recipient phone numbers might be incorrect if contact was updated
4. Messages showed outdated content in webapp preview
5. No way to use `translation_key` pattern consistently across all messages

## Root Cause

Message generation happened synchronously during report processing. The `messages` array was created once and stored, never updated before sending. This design assumed data wouldn't change, which was incorrect for real-world deployments.

## Solution

Implemented "just-in-time" message generation:

1. **Deferred message generation**: Messages are now generated just before being marked ready to send, not when report is received
2. **Created shared library**: Extracted message sending code into shared library used by both sentinel and webapp
3. **Updated webapp rendering**: Messages are generated when displayed in webapp so users see current content
4. **Enabled translation_key pattern**: All messages can now use `translation_key` instead of hardcoded `messages` array
5. **Dynamic data resolution**: Messages resolve translation values, context (doc, patient), and recipient phone numbers at send time
6. **Refactored report rendering**: Moved code from `/packages` into Angular service to access shared library

## Code Patterns

- Use shared libraries for code used across multiple services (sentinel + webapp)
- Pattern: Generate messages dynamically using current data instead of static stored values
- Pattern: `sentinel/lib/messages.js` contains message generation logic
- Pattern: `sentinel/lib/schedules.js` handles message scheduling
- Use `translation_key` pattern for all messages for consistency
- Resolve data dynamically: `getMessage(msg, doc, patient, contact)` at send time
- Handle errors gracefully: `utils.addError(doc, { message: e.message + ': ' + messageConfig.translation_key || messageConfig.messages })`

## Design Choices

Chose just-in-time generation over periodic regeneration because:
- Only generates messages that will actually be sent (performance)
- Always uses most current data (accuracy)
- Reduces storage (don't store intermediate message states)
- Simplifies data model (messages computed, not stored)

Created shared library because:
- Both sentinel and webapp need to generate messages
- Ensures consistent message generation logic
- Easier to maintain single source of truth
- Enables webapp to show accurate message previews
