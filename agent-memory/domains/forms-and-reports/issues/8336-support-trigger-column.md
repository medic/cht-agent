## Problem

ODK added support for a `trigger` column in XLSForms years ago, enabling two useful patterns: setting dynamic defaults on fields and triggering calculations when a value changes. The CHT did not support this because cht-conf used an older version of pyxform that did not generate the required `xforms-value-changed` event in the XForm output.

## Root Cause

The version of pyxform bundled with cht-conf was too old to convert the `trigger` column into the corresponding XForm event bindings. Even though Enketo Core (from version 5.18.0) supported the `xforms-value-changed` event, the XForm XML generated by cht-conf never included it.

## Solution

Uplifted cht-conf to use a newer version of pyxform that supports the `trigger` column. This was a large change (PR #10269, 78 files) because the pyxform upgrade also affected how XLSForms are converted to XForm XML, requiring updates to many form fixtures and test expectations.

## Code Patterns

- The `trigger` column in XLSForm maps to the `xforms-value-changed` event in XForm XML
- Dynamic defaults: set a field's default value based on other fields, evaluated once when the field becomes relevant
- Value-change calculations: trigger a recalculation when a specific field value changes, unlike regular calculations that fire on any change
- File: `webapp/src/js/enketo/widgets.js` is the widget registry where Enketo features are enabled
- Pattern: when uplifting form processing dependencies, expect cascading changes across form fixtures in tests
- See ODK docs for trigger usage: https://docs.getodk.org/form-logic/#dynamic-defaults

## Design Choices

- Uplifted pyxform in cht-conf rather than implementing custom trigger parsing, to stay aligned with the ODK ecosystem
- Accepted the large test fixture update cost because it was a one-time migration
