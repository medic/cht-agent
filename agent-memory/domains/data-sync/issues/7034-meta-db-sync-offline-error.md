## Problem

When an offline user attempted synchronization, the meta database replication could fail and generate a feedback document containing an empty error message.

These feedback documents provided little diagnostic value and created unnecessary noise for developers investigating issues.

## Root Cause

The issue was caused by the use of `PouchDB.sync()` for meta database synchronization.

When replication failed while the device was offline, the rejection generated by `PouchDB.sync()` was not surfaced correctly through Zone.js. Zone.js interpreted the failure as an uncaught promise rejection and generated an error object with an empty message (`{ name: 'Error', message: '' }`).

The feedback service then created feedback documents for these expected offline failures, resulting in large numbers of useless reports.

## Solution

The fix replaced the use of `PouchDB.sync()` with explicit replication calls:

* `local.replicate.to(remote)`
* `local.replicate.from(remote)`

wrapped in `Promise.all(...)` with proper error handling.

This prevented Zone.js from generating uncaught promise rejections during expected offline replication failures and eliminated the creation of unnecessary feedback documents.

The change also made replication error handling more explicit and predictable.

## Code Patterns

* Expected offline failures should not generate unnecessary feedback reports.
* Be careful when using `PouchDB.sync()` with promise-based error handling.
* Zone.js can surface replication failures differently than expected.
* Explicit replication calls may provide more predictable error handling than combined sync operations.

## Design Choices

* Replaced `PouchDB.sync()` instead of adding workarounds around Zone.js behavior.
* Treated offline replication failures as expected operational conditions.
* Reduced noise in feedback reporting systems.
* Preferred explicit replication control and error handling.
